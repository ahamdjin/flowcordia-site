# Flowcordia

Flowcordia is a Git-native workflow platform where visual builders and developers work on the same reviewed workflow.

The canvas is not a separate source of truth. A workflow keeps one identity across its visual model, canonical document, generated source, Git history, deployment version, and run history.

## The operating loop

<Steps>

<Step>Build visually</Step>

Compose triggers, conditions, transformations, and tasks in Studio.

<Step>Extend with typed code</Step>

Use repository-owned functions when a workflow needs behavior that should remain explicit code.

<Step>Review through Git</Step>

Publish the workflow document, generated task source, and governed source changes together as one proposal.

<Step>Run the reviewed version</Step>

Preview and production execution remain tied to an exact commit and deployment version.

</Steps>

## What Flowcordia protects

| Boundary | Contract |
| :-- | :-- |
| Workflow identity | The same workflow ID follows the canvas, source, proposal, deployment, and run. |
| Git history | Production changes remain reviewable, attributable, and reversible. |
| Secrets | Workflows reference secret names; secret values do not enter workflow documents. |
| Exact execution | A changed proposal head is treated as a changed executable version. |
| Unsupported code | Code that cannot be represented safely remains code instead of becoming a misleading graph. |

## Current status

Flowcordia is under active development in public. The Studio, compiler, Git proposal lifecycle, typed-function contract, and exact-preview path are being developed as connected product foundations rather than isolated demonstrations.
